# 코드 아키텍트 스킬 (Code Architect Skill)

> **담당:** 개발부
> **Tier:** 2 (Claude Sonnet)
> **Skill ID:** `code`

## Purpose

SPEC.md를 입력받아 FSD(Feature-Sliced Design) 패턴을 준수하는 보일러플레이트 코드를 생성한다.
아키텍처 결정 기록(ADR)을 자동 작성하고, 기존 코드베이스의 패턴 준수 여부를 검사한다.

## Trigger

- 새로운 피처 개발 시작 시
- `@code architect {feature_name}` 명령 호출 시
- PR 생성 시 자동 패턴 검사

## Input

```json
{
  "mode": "generate | audit | adr",
  // generate: 새 기능의 기본 코드 구조 자동 생성
  // audit: 기존 코드가 규칙을 잘 따르고 있는지 검사
  // adr: 이번 개발에서 내린 설계 결정을 문서로 기록

  "spec_path": "기능 설명서(SPEC.md) 파일의 경로",
  "feature_name": "만들 기능의 이름 (예: user-auth, payment)",

  "layer": "app | pages | widgets | features | entities | shared",
  // FSD 아키텍처의 레이어 구분
  // app: 앱 전체 설정  |  pages: 화면 단위
  // widgets: 여러 화면에서 재사용하는 큰 블록
  // features: 특정 기능 단위  |  entities: 데이터 모델
  // shared: 어디서든 쓰는 공통 유틸

  "framework": "nextjs | react-native-expo | nestjs",
  // 어떤 기술 스택으로 만들지

  "options": {
    "include_tests": true,      // 테스트 파일도 같이 만들지 여부
    "include_stories": false,   // UI 컴포넌트 문서(Storybook)도 만들지 여부
    "include_api_route": true,  // API 엔드포인트 파일도 만들지 여부
    "strict_types": true        // any 타입 사용을 금지하고 모든 타입을 명시할지 여부
  }
}
```

## Output

### mode: generate

**디렉토리 구조:**
```
src/features/{feature_name}/
├── api/
│   ├── {feature_name}.api.ts        # 서버와 통신하는 함수들 (데이터 가져오기/보내기)
│   └── {feature_name}.api.test.ts   # API 함수가 제대로 동작하는지 확인하는 테스트
├── model/
│   ├── {feature_name}.model.ts      # 화면 상태 관리 (Zustand — 전역 상태를 담는 저장소)
│   └── {feature_name}.types.ts      # 이 기능에서 쓰는 데이터 형태 정의
├── ui/
│   ├── {FeatureName}.tsx            # 실제 화면에 보이는 컴포넌트
│   └── {FeatureName}.test.tsx       # 화면이 제대로 렌더링되는지 테스트
├── lib/
│   └── {feature_name}.utils.ts      # 이 기능에서만 쓰는 계산/변환 함수들
└── index.ts                         # 외부에서 이 기능을 가져다 쓸 때의 진입점
                                     # (이 파일을 통해서만 외부 접근 허용 — 캡슐화)
```

**파일 최상단 주석 템플릿:**
```typescript
/**
 * 이 파일이 속한 기능: {feature_name}
 * FSD 레이어: features (기능 단위 코드)
 * 연관 기획서: SPEC.md의 {섹션명} 참고
 * 최초 생성일: 2026-03-17
 */
```

### mode: audit

```json
{
  "score": 85,  // 코드 품질 점수 (100점 만점). 85점 = 상위 15% 수준의 문제 있음

  "violations": [
    {
      "rule": "no-cross-layer-import",
      // 위반 규칙 이름: 레이어 간 직접 참조 금지
      // (features가 widgets을 직접 불러오면 안 됨 — 의존성이 꼬이기 때문)

      "file": "src/features/auth/ui/LoginForm.tsx",
      "line": 3,  // 문제가 발생한 줄 번호

      "message": "features 레이어에서 widgets 레이어를 직접 import할 수 없습니다.",
      "severity": "error",  // error: 반드시 고쳐야 함 | warning: 권장 수정
      "suggestion": "shared 레이어를 통해 접근하세요."
      // 어떻게 고쳐야 하는지 AI가 직접 제안
    }
  ],

  "summary": {
    "total_files": 42,   // 전체 검사한 파일 수
    "compliant": 38,     // 규칙을 잘 따르는 파일 수
    "violations": 3,     // 규칙 위반 파일 수
    "warnings": 1        // 경고 수준의 파일 수
  }
}
```

### mode: adr

```markdown
# ADR-{번호}: {결정 제목}
<!-- ADR: Architecture Decision Record — 왜 이렇게 만들었는지 기록하는 문서 -->
<!-- 나중에 "왜 이렇게 짰지?" 라는 질문에 답할 수 있게 남겨두는 것 -->

## Status: Accepted  <!-- 이 결정이 채택됨 -->
## Date: 2026-03-17

## Context
<!-- 이 결정을 내리게 된 배경과 문제 상황 (SPEC.md에서 자동 추출) -->

## Decision
<!-- 우리가 선택한 방향과 그 이유 -->

## Consequences
### Positive   <!-- 이 결정으로 얻는 것 -->
- ...
### Negative   <!-- 이 결정으로 생기는 단점/트레이드오프 -->
- ...
### Risks      <!-- 앞으로 주의해야 할 위험 요소 -->
- ...
```

## Process

```
1. SPEC.md 파싱 → 요구사항 추출
   // 기획서를 읽어서 "무엇을 만들어야 하는지" 파악

2. mode별 분기
   ├─ generate:
   │   ├─ FSD 레이어 결정 (feature_name + layer 기반)
   │   ├─ 보일러플레이트 생성 (framework별 템플릿 적용)
   │   │   // 반복적으로 작성해야 하는 기본 코드 구조를 자동으로 만들어줌
   │   ├─ 파일 최상단 주석 자동 삽입 (기능/레이어/기획서 연결)
   │   ├─ Public API(index.ts) 자동 구성
   │   │   // 외부에서 이 기능을 쓸 때 어떤 것들을 제공할지 정리
   │   └─ 테스트 스켈레톤 생성 (include_tests 옵션 시)
   │       // 테스트 파일의 빈 껍데기를 만들어서 개발자가 내용만 채우면 되게 함
   │
   ├─ audit:
   │   ├─ import 경로 분석 → 레이어 간 의존성 그래프 생성
   │   │   // 어떤 파일이 어떤 파일을 가져다 쓰는지 지도를 그림
   │   ├─ FSD 규칙 위반 탐지 (레이어 간 잘못된 참조)
   │   ├─ `any` 타입 사용 탐지 (타입 안전성 훼손하는 코드)
   │   ├─ dead code 탐지 (어디서도 안 쓰는 코드)
   │   └─ 점수 산출 (100점 만점)
   │
   └─ adr:
       ├─ SPEC.md 컨텍스트 추출
       ├─ 기존 ADR 히스토리 참조 (일관성 유지)
       └─ 결정문 초안 생성

3. strict_types 옵션 시 모든 함수/변수에 명시적 타입 필수
   // any 대신 정확한 타입을 써야 → 버그가 코드 작성 시점에 바로 발견됨

4. 결과물 SPEC.md에 역참조 링크 추가
   // 기획서에서 이 코드를 바로 찾아갈 수 있게 링크 추가
```

## Constraints

- `any` 타입 사용 절대 금지 (strict_types: true 기본값) `[ENG]`
  // any는 "이 데이터가 뭔지 모르겠다"는 뜻 — 버그의 온상
- 배럴 파일(index.ts)에서만 외부 노출 `[ENG]`
  // 기능의 내부 구조는 숨기고, 필요한 것만 선택적으로 공개
- 순환 의존성(circular dependency) 감지 시 즉시 에러 `[ENG]`
  // A가 B를 쓰고, B가 A를 쓰면 무한루프 → 앱이 뻗음
- 한 파일 최대 300줄 (초과 시 분리 제안) `[DRY]`
  // 파일이 너무 길면 이해하기 어려워짐

## Skill Chaining

- **downstream:** Edge-case Hunter Skill → 자동 체이닝 (generate 모드 완료 후)

## Hard Gates

| ID | 조건 | FAIL 기준 |
|----|------|-----------|
| G1 | 타입 안전성 | `any` 남용 또는 타입 오류 가능성이 명확함 |
| G2 | 아키텍처 경계 | 제품별 아키텍처 규칙(FSD/NestJS 모듈)을 위반함 |
| G3 | 검증 가능성 | 타입체크, 테스트, 수동 검증 중 하나도 제시하지 않음 |

## Soft Findings

| ID | 조건 | 조치 |
|----|------|------|
| S1 | 중복 가능성 | 기존 모듈/훅/컴포넌트 재사용 후보를 표시 |
| S2 | 테스트 보강 | 위험도가 높은 분기부터 테스트 제안 |
