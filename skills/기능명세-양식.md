# 기능명세 양식 (SPEC Template)

> **사용법:** 새 기능을 시작할 때 이 파일을 복사해 `{기능명}/명세서.md`로 저장한다.
> 리서치·계획·구현 단계 모두 이 파일을 기준으로 삼는다.
> 클로드가 세션 시작 시 가장 먼저 읽는 파일이다.

---

## 복사 후 사용 경로

```
features/
└── {기능명}/
    ├── 명세서.md     ← 이 양식을 복사해서 씀
    ├── 리서치.md     ← IRESEARCH 실행 후 자동 생성
    ├── 계획.md       ← IPLAN 실행 후 자동 생성
    └── ... (실제 코드)
```

# [기능 이름] 명세서

> **작성자:** KK / SY / IPAI
> **작성일:** YYYY-MM-DD
> **상태:** 초안 | 검토 중 | 확정 | 구현 완료

---

## 1. 왜 만드는가 (목적)

> 이 기능이 없으면 어떤 문제가 생기는지 한 문장으로.

```
예시: 사용자가 결제 후 영수증을 받지 못해 환불 문의가 반복되고 있다.
```

---

## 2. 누가 쓰는가 (사용자)

```
주 사용자: 예) 결제를 완료한 로그인 유저
보조 사용자: 예) 관리자 (영수증 재발송 기능)
```

---

## 3. 어떻게 쓰는가 (사용자 흐름)

> 클릭 → 입력 → 결과 순서로 단계를 적는다. 분기도 포함.

```
1. 사용자가 결제 완료 페이지에서 "영수증 보기" 버튼 클릭
2. 이메일로 영수증 발송 + 화면에 미리보기 표시
3. "다운로드" 버튼 클릭 시 PDF 저장
   └─ 실패 시: "잠시 후 다시 시도해주세요" 토스트 + 재시도 버튼
```

---

## 4. 범위

### 이번에 하는 것 ✅

```
- 이메일 영수증 자동 발송
- 영수증 미리보기 (웹)
- PDF 다운로드
```

### 이번에 하지 않는 것 ❌

> 명시적으로 선 긋는 것이 중요하다 — 클로드가 범위를 넘지 않도록.

```
- 영수증 템플릿 커스터마이징 (다음 스프린트)
- 세금계산서 발행 (별도 기능으로 분리)
- 카카오 알림톡 발송 (ElevenLabs 연동 후 검토)
```

---

## 5. 성공 기준 (언제 완료인가)

> 테스트 가능한 기준으로 작성한다. "잘 작동한다"는 기준이 아님.

```
[ ] 결제 완료 후 5초 이내 이메일 수신
[ ] 영수증 미리보기가 모바일에서도 깨지지 않음
[ ] PDF 다운로드 파일명: `영수증_YYYYMMDD_주문번호.pdf`
[ ] 발송 실패 시 n8n 알림 발송 + 재시도 큐 등록
```

---

## 6. 제약조건 (건드리면 안 되는 것)

> 리서치 단계에서 이 목록을 기준으로 위험 지점을 체크한다.

```
- checkout.ts 서버 액션은 수정 금지 [ENG] — 다른 결제 플로우와 공유 중
- 이메일 발송은 기존 sendEmail() 유틸 재사용 [DRY]
- 영수증 디자인은 브랜드-아이덴티티 토큰만 사용 [PROD]
- 개인정보(이름, 이메일)는 서버에서만 처리, 클라이언트 노출 금지 [SEC]
```

---

## 7. 기술 결정사항

> 이미 결정된 것만 적는다. 미결은 계획.md에서 결정.

```
- 이메일 서비스: Resend API (기존 사용 중)
- PDF 생성: @react-pdf/renderer (신규 추가 필요 — SY 승인 필요)
- 상태관리: 기존 Zustand store 확장
```
