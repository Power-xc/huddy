## 규칙 강제 등급

> 모든 규칙은 아래 세 등급 중 하나로 표시된다.

| 등급 | 의미 |
|------|------|
| `MUST` | 무조건 지킨다. 예외 없음. 어기면 코드 반려. |
| `SHOULD` | 특별한 이유가 없으면 지킨다. 어길 경우 이유를 주석으로 명시. |
| `SHOULD NOT` | 특별한 이유가 없으면 하지 않는다. 할 경우 이유를 주석으로 명시. |

## 0. 세션 시작 프로토콜

> `[IPAI][MUST]` 새 채팅(세션)을 열면 코드부터 짜지 않는다.
> 아래 순서대로 맥락을 복원한 뒤 작업을 시작한다.
> 오토 컴팩션이 와도 이 파일들은 살아남는다.

### 0-1. 반드시 읽는 파일 (순서대로)

```
1순위 — 행동 규범
   └─ skills/개발/운영-프로토콜.md (지금 이 파일)

2순위 — 진행 중인 작업 (있을 경우)
   └─ {기능명}/명세서.md   → 이 기능이 무엇인지
   └─ {기능명}/리서치.md   → 코드베이스에서 발견한 것들
   └─ {기능명}/계획.md     → 구현 계획 + 체크리스트 현황

3순위 — 프로젝트 전체 구조 (처음 세션이라면)
   └─ SPEC.md 또는 README.md (프로젝트 루트)
```

### 0-2. 세션 시작 시 클로드 자기소개 형식

> 파일을 읽은 뒤 아래 형식으로 현재 상태를 보고한다. 바로 작업 시작 금지.

```
"세션 복원 완료.

현재 작업: {기능명} 구현 중
진행 상태: 계획.md 체크리스트 {X}/{전체}개 완료
다음 작업: {체크리스트 중 첫 번째 미완료 항목}
주의 사항: {리서치.md에서 발견한 위험 지점 요약}

계속 진행할까요?"
```

> 진행 중인 작업이 없을 경우:
```
"세션 복원 완료. 운영-프로토콜 V6.1 + 실행 스킬 26종 + 운영 문서/템플릿 5종 + 감사표 로드됨.
무엇을 만들까요? (명세서.md 작성부터 시작 권장)"
```

### 0-3. CLAUDE.md 설정 권장

> 프로젝트 루트에 `CLAUDE.md` 파일을 두면 세션 시작 시 자동으로 로드된다.

```markdown
# CLAUDE.md (프로젝트 루트에 위치)

## 필수 로드
- skills/개발/운영-프로토콜.md — IPAI 행동 규범 (항상 준수)

## 현재 진행 작업
- {기능명}/명세서.md
- {기능명}/계획.md

## 핵심 제약
- 아키텍처 변경, 라이브러리 추가는 사령관 승인 필수
- 기능 추가 전 반드시 명세서.md 확인
```

## 1. 절대 원칙

> `[IPAI][MUST]` 어떤 상황에서도 예외 없이 지킨다. 사령관의 명시적 허가 없이 아래 원칙을 어길 수 없다.

| 원칙 | 등급 | 규칙 |
|------|------|------|
| **바이브 코딩 금지** | `MUST` | 전체 맥락 없이 짐작으로 코드를 짜지 않는다. 코딩 전 반드시 기존 토큰, 컴포넌트 구조, `SPEC.md`를 먼저 확인한다. |
| **모르면 질문하라 (Fail-fast)** | `MUST` | 아키텍처, 스키마, 라이브러리 추가 등 기술 판단이 불확실한 경우 절대 스스로 가정(Assume)하지 말고 사령관(인간)에게 멈춰서 질문한다. `[IPAI]` |
| **중복 개발 금지** | `MUST` | 새 UI·로직을 만들기 전 `@/components/shared` 및 `hooks` 폴더를 먼저 탐색한다. 비슷한 것이 있으면 만들지 않고 재사용한다. `[DRY]` |
| **한국어 주석 필수** | `MUST` | 변수명·코드는 영어로, 설계 의도·제약·"왜 이렇게 짰는지"는 반드시 한국어(JSDoc 스타일)로 남긴다. `[IPAI]` |
| **SPEC 선행 확인** | `MUST` | 기능을 수정하기 전 반드시 연관된 `SPEC.md` 섹션을 읽는다. `[SPEC]` |
| **단독 판단 금지** | `MUST` | 아키텍처 변경, 라이브러리 추가, DB 스키마 수정은 사령관 승인 없이 단독으로 실행하지 않는다. |
| **교체 원칙** | `MUST` | 새 구현이 기존 것을 대체하면 기존 코드를 완전히 삭제한다. 하위 호환 shim, 더미 래퍼, deprecated 주석을 남기지 않는다. `[ENG]` |
| **기존 코드 보존** | `MUST` | 작동하는 코드는 명시적 지시 없이 건드리지 않는다. 필요한 곳만 수정한다. `[IPAI]` |

## 2. 코드 품질 기준

> `[ENG]` 수치로 측정 가능한 품질 기준. 아래 임계치를 넘으면 코드 스멜(Code Smell)로 간주하고 즉시 리팩터링한다.

### 2-1. 하드 리밋 (Hard Limits)

| 항목 | 임계치 | 초과 시 조치 |
|------|--------|------------|
| 함수 길이 | **100줄** | 기능 단위로 분리 |
| 순환 복잡도 | **8** | 조건문 단순화 또는 분기 분리 |
| 함수 파라미터 수 | **5개** | 객체로 묶기 |
| 파일 길이 | **300줄** | 파일 분리 |
| 중첩 조건문 깊이 | **2단계** | 조기 반환(early return) 패턴 적용 |
| 클래스 공개 메서드 수 | **5개** | 책임 분리 |

### 2-2. Zero Warnings 정책

`[ENG][MUST]` 린터·타입체커·빌드 도구의 경고를 **단 하나도** 남기지 않는다.
진짜 고칠 수 없는 경고라면 인라인 ignore와 함께 **반드시 이유를 한국어로 주석**한다.

```typescript
// [ENG] Supabase 자동생성 타입과 충돌 — 수동 수정 불가, 다음 스키마 재생성 시 해결 예정
// eslint-disable-next-line @typescript-eslint/no-explicit-any
const response = supabaseClient.rpc('custom_function') as any;
```

### 2-3. 코드 스멜 탐지 체크리스트

함수를 작성한 후 아래 항목을 스스로 검토한다.

```
[ ] 가독성 — 이름만 보고 무엇을 하는지 바로 알 수 있는가?
[ ] 복잡도 — 순환 복잡도(분기 수)가 8을 넘지 않는가?
[ ] 파라미터 — 5개 이하인가? 초과 시 객체로 묶었는가?
[ ] 불필요한 추출 — 이 함수가 재사용되거나, 독립 테스트가 필요하거나,
                    가독성을 크게 높이는가? 그렇지 않으면 인라인으로 둔다.
[ ] 숨겨진 의존성 — 파라미터가 아닌 외부 상태에 의존하고 있지 않은가?
[ ] 테스트 가능성 — 외부 의존성 없이 단독으로 테스트할 수 있는가?
[ ] 이름 최적화 — 3가지 대안 이름을 떠올려보고 가장 명확한 것을 선택했는가?
[ ] any 타입 — `any`를 사용하지 않았는가?
```

## 3. 아키텍처 전략 — SY 규칙

> `[ENG]` UI와 로직을 완전히 분리해 어떤 부품이든 독립적으로 교체·재사용 가능하게 만든다.

### 3-1. 로직과 UI 완전 분리 `[ENG][MUST]`

- **UI 컴포넌트** → 오직 화면을 그리고 Props를 받는 역할만. 판단·계산이 없는 "멍청한(Dumb)" 컴포넌트로 유지한다.
- **비즈니스 로직** → 10줄을 넘어가면 무조건 `use[기능명].ts` Custom Hook으로 분리한다.

```typescript
// ─────────────────────────────────────────────────────────────
// ❌ [ENG] 위반: UI 안에 로직이 뒤섞여서 재사용도, 테스트도 불가능
// ─────────────────────────────────────────────────────────────
function PaymentButton() {
  const [loading, setLoading] = useState(false);
  const handleClick = async () => {
    setLoading(true);
    await fetch('/api/pay', { method: 'POST' }); // UI가 직접 API를 부름 — 금지
  };
  return <button onClick={handleClick}>결제</button>;
}

// ─────────────────────────────────────────────────────────────
// ✅ [ENG][DRY] 로직과 UI 분리 — 양쪽 모두 독립적으로 테스트·재사용 가능
// ─────────────────────────────────────────────────────────────

// usePayment.ts — 결제에 관한 모든 로직을 여기서만 관리
function usePayment() {
  const [loading, setLoading] = useState(false);
  const pay = async () => { /* 결제 로직 전부 여기 */ };
  return { loading, pay };
}

// PaymentButton.tsx — 화면을 그리는 것에만 집중
function PaymentButton() {
  const { loading, pay } = usePayment();
  return <button onClick={pay} disabled={loading}>결제</button>;
}
```

### 3-2. 레고 블록화 (Atomic Design) `[ENG][SHOULD]`

```
atoms/      → Button, Input, Icon, Badge
             (더 이상 쪼갤 수 없는 가장 작은 단위)
molecules/  → SearchBar = Input + Button
organisms/  → Header = Logo + SearchBar + UserMenu
pages/      → 실제 라우트에 매핑되는 완성된 화면
```

### 3-3. 배럴 엑스포트 (Barrel Export) `[ENG][MUST]`

새 컴포넌트를 만들면 **즉시** 상위 `index.ts`에 등록한다.

```typescript
// components/ui/index.ts — 모든 UI 컴포넌트의 단일 출입구
export * from './Button';
export * from './Input';
export * from './Modal';

// ❌ 경로를 직접 찾아가는 방식 — 경로가 바뀌면 전부 깨짐
import { Button } from '@/components/ui/Button/Button';

// ✅ [ENG] 배럴을 통해 가져오는 방식 — 경로 변경에 안전
import { Button, Input, Modal } from '@/components/ui';
```

### 3-4. FSD 레이어 의존성 규칙 `[ENG][MUST]`

```
app → pages → widgets → features → entities → shared

// [ENG] 상위 레이어만 하위를 import 가능. 역방향은 순환 의존성을 만들어 시스템 전체가 꼬임
// ❌ 금지: features 파일 안에서 widgets을 직접 import
import { SomeWidget } from '@/widgets/dashboard';
```

### 3-5. 의존성 최소화 `[ENG][SHOULD NOT]`

새 라이브러리는 명시적 요청 또는 정당한 이유 없이 추가하지 않는다.
모든 의존성은 공격 표면(attack surface)이자 유지보수 부담이다.

```typescript
// [ENG] 날짜 포맷팅을 위해 date-fns 전체를 추가하는 것은 과도함
// 단순 포맷이라면 Intl.DateTimeFormat (브라우저 내장 API)으로 처리
const formatted = new Intl.DateTimeFormat('ko-KR').format(new Date());
```

## 5. 실행 워크플로우

> `[IPAI][MUST]` 지시를 받으면 코드를 **바로 짜지 않는다**.
> 반드시 아래 4단계를 따른다. 각 단계는 실제 파일로 산출물을 남긴다.
> 채팅창 요약으로 끝내는 것은 금지 — 오토 컴팩션이 와도 파일은 살아남는다.

---

### STEP 1 — 리서치 `[MUST]`

> **목적:** 무지한 변경을 막는다.
> 코드베이스를 깊이 읽고, 발견한 내용을 **`리서치.md`** 파일로 작성한다.

```
산출물: {작업명}/리서치.md

포함 내용:
├─ 관련 파일 목록 + 각 파일의 역할
├─ 현재 아키텍처 동작 방식 (코드 흐름 추적)
├─ 재사용 가능한 기존 부품 (중복 생성 방지)
├─ 건드리면 안 되는 영역 (사이드이펙트 위험 지점)
├─ ORM 관례 / API 패턴 / 레이어 규칙 확인
└─ 리서치 중 발견한 기존 버그·부채

금지:
- 채팅창에 구두 요약으로 끝내기
- 파일을 안 열어보고 추측으로 분석하기
- 관련 없어 보이는 파일을 건너뛰기
```

**효과:** 기존 레이어 무시 함수, ORM 관례 어기는 마이그레이션, 중복 API 엔드포인트를 이 단계에서 사전 차단한다.

---

### STEP 2 — 계획 `[MUST]`

> **목적:** 잘못된 변경을 막는다.
> 구현 방법을 **`계획.md`** 파일로 상세 작성한다. 아직 코드는 한 줄도 짜지 않는다.

```
산출물: {작업명}/계획.md

포함 내용:
├─ 접근 방식 상세 설명 (왜 이 방식인지 근거 포함)
├─ 파일 경로 + 변경/생성 파일 목록
├─ 핵심 코드 스니펫 (실제 구현 전 설계도)
├─ 트레이드오프 (이 방식의 장단점, 포기한 대안)
├─ 단계별 체크리스트 ([ ] 구현 → [ ] 타입체크 → [ ] 등록)
└─ KK/SY 권한 영역 구분 명시

형식 예시:
## 접근 방식
결제 로직을 usePayment Hook으로 통합. [ENG]
기존 useCart와 인터페이스를 맞춰 재사용성 확보. [DRY]

## 파일 목록
- 생성: src/features/payment/hooks/usePayment.ts
- 생성: src/features/payment/ui/PaymentButton.tsx
- 수정: src/features/payment/index.ts (배럴 등록)

## 트레이드오프
- 장점: 결제 로직 중앙화, 테스트 용이
- 단점: features/payment에 의존성 추가

## 체크리스트
- [ ] usePayment.ts 구현
- [ ] PaymentButton.tsx 구현
- [ ] index.ts 배럴 등록
- [ ] 타입체크 통과
```

---

### STEP 3 — 주석 달기 `[MUST]`

> **목적:** 창의적 판단과 방향 조정을 이 단계에서 완료한다.
> 사령관이 `계획.md`를 에디터에서 직접 검토하고 인라인 메모를 추가한다.
> 클로드는 메모를 반영해 `계획.md`를 업데이트한다. **아직 구현하지 않는다.**

```
메모의 역할:
├─ 가정 수정      예: "이 부분 틀렸어, DB 스키마가 달라"
├─ 접근 방식 거부  예: "Hook 말고 Zustand store로"
├─ 제약조건 추가   예: "이 함수는 서버 컴포넌트에서만 써야 해"
└─ 도메인 지식 전달 예: "이 API는 rate limit이 있어서 캐싱 필수"

방향 조정 패턴:
├─ 좋은 건 취하고 나머진 버리기
├─ 과감하게 쳐내기 (scope 줄이기)
├─ 건드리면 안 되는 선 긋기
└─ 기술 선택은 사령관이 결정

메모 크기: 딱 두 단어도 충분 ("Zustand로", "빼줘", "서버만")

클로드 응답 형식:
"메모 [X]개 반영 완료. 계획.md 업데이트했습니다.
 아직 구현하지 않았습니다. 구현 지시를 주세요."
```

> 이 단계에서 창의적 작업과 중요한 판단이 대부분 끝난다.
> 이 다음부터는 기계적으로 실행만 한다.

### STEP 4 — 구현 `[MUST]`

> **목적:** 이미 내려진 결정을 방해 없이 실행한다.
> 모든 결정은 `계획.md`에서 완료된 상태다. 새로운 창의적 판단 없이 코딩한다.

```
실행 규칙:
├─ 계획.md 체크리스트 순서대로 구현
├─ 작업·단계 완료 시 즉시 [x]로 표시
├─ 모든 체크리스트가 [x]가 될 때까지 멈추지 않는다
├─ any / unknown 타입 사용 금지 [ENG]
└─ 각 파일 저장 후 즉시 타입체크 실행 (새 에러 발생 시 즉시 수정)

완료 보고 형식:
"구현 완료.
 ✅ usePayment.ts — 타입체크 통과
 ✅ PaymentButton.tsx — 타입체크 통과
 ✅ index.ts 배럴 등록
 계획.md 전체 체크리스트 완료."

금지:
- 구현 중 새로운 방향 제안
- 계획.md에 없는 파일 임의 생성
- "이렇게 바꾸는 게 어떨까요?" 식의 중간 개입
```

### 3단계 방어체계 요약

```
STEP 1 리서치  → 무지한 변경 차단 (모르고 건드리는 것 방지)
STEP 2 계획    → 잘못된 변경 차단 (알고도 틀린 방향 방지)
STEP 3 주석    → 방향 조정 완료   (구현 전 모든 판단 확정)
STEP 4 구현    → 방해 없이 실행   (결정 완료 상태에서 기계적 코딩)
```

## 6. IPAI 커맨드

> 사령관이 한 단어로 품질 게이트를 발동할 수 있는 단축 커맨드.
> 커맨드를 입력하면 AI가 해당 체크리스트를 전부 수행하고 결과를 보고한다.

| 커맨드 | 단계 | 동작 |
|--------|------|------|
| `IRESEARCH` | STEP 1 | 코드베이스 심층 분석 → `리서치.md` 생성. 관련 파일 전수 탐색, 아키텍처 흐름 추적, 재사용 부품 발굴, 위험 지점 식별 |
| `IPLAN` | STEP 2 | `리서치.md` 기반 구현 계획 수립 → `계획.md` 생성. 파일 경로·스니펫·트레이드오프·체크리스트 포함. 코드 미작성 |
| `IEXEC` | STEP 4 | `계획.md` 체크리스트 전부 구현. 완료 항목 즉시 [x] 표시. 전체 완료까지 중단 없음. 타입체크 연속 실행 |
| `ICODE` | STEP 4 후 | 코드 작성 후 체크 — 린터, 타입체크, Zero Warnings 확인 |
| `ICHECK` | 전체 | 전체 품질 검토 — 코드 스멜, 하드 리밋, 주석, 에러 처리, 보안 취약점 |
| `IDESIGN` | 전체 | 디자인 무결성 검토 — 하드코딩된 값, 미구현 인터랙티브 상태, 반응형 누락 `[PROD]` |
| `ISEC` | 전체 | 보안 검토 — 입력값 검증, 인증·인가, 환경변수 노출, XSS/SQL injection `[SEC]` |
| `ICOST` | 전체 | 비용 검토 — Tier 적절성, API 호출 횟수, 캐싱 기회 `[COST]` |
| `IDOCS` | 커밋 전 | 명세서/문서 동기화 — 코드 수정 후 과거 리서치 문서/계획서가 뒤처지지 않게 실제 최신 코드 기준으로 갱신 `[ENG]` |
| `IGIT` | 마지막 | 커밋 준비 — 불필요한 코드 제거, 커밋 메시지 작성 |

**표준 워크플로우:**
```
사령관: "결제 기능 만들어줘"

AI: IRESEARCH 실행 → 리서치.md 작성 완료
    "리서치 결과: useCart 재사용 가능, /api/payment 엔드포인트 이미 존재
     건드리면 안 되는 부분: checkout.ts 서버 액션 (side effect 있음)
     계획.md 작성할까요?"

사령관: "응"

AI: IPLAN 실행 → 계획.md 작성 완료
    "계획.md 에디터에서 검토 후 메모 달아서 보내주세요.
     아직 구현 안 했습니다."

사령관: [계획.md에 메모 추가 후 전달]
        "// Zustand로 바꿔줘" "// 이 파일은 건드리지 마"

AI: 메모 3개 반영 → 계획.md 업데이트 완료
    "아직 구현 안 했습니다. IEXEC 입력하면 시작합니다."

사령관: "IEXEC"

AI: [체크리스트 순서대로 구현 → 완료 항목 [x] 표시 → 타입체크]
    "구현 완료. 계획.md 전체 체크리스트 [x]."
```

### 6-2. 스킬 체이닝 아웃풋 표준 `[MUST]`

모든 스킬 분석·생성 결과(Output) 문서는 최하단에 기계가 파싱할 수 있는 **[표준 JSON Payload]** 블록을 포함해야 한다.
- **목적:** 다음 스킬이 장황한 마크다운을 다시 읽지 않고, 요약된 JSON을 입력값으로 받아 곧바로 다음 행동(체이닝)을 이어나갈 수 있게 하기 위함.
- **예시:** `{"action": "@strategy analyze", "bottleneck": "MOFU"}`

#### 최소 필드

모든 실행 스킬의 하단 Registry Metadata 또는 표준 JSON Payload에는 아래 필드를 포함한다. 새 스킬은 `skills/SKILL_TEMPLATE.md`를 기준으로 작성한다.

```json
{
  "skill_id": "스킬 고유 ID",
  "skill_name": "사람이 읽는 스킬 이름",
  "owner": "담당 부서",
  "tier": "1 | 2",
  "trigger_type": "manual | scheduled | webhook | pre-deploy",
  "dependencies": [],
  "last_updated": "YYYY-MM-DD"
}
```

보안 게이트형 스킬은 실행 결과 Output에 `verdict`, `hard_gates`, `soft_findings`, `summary`를 추가로 포함한다.
운영 추적이 필요한 실행 결과는 `skills/OPERATING_LOOP.md`의 Action Payload를 따라 `token → insight → task` 루프로 연결한다.

## 7. 코드 주석 규칙

> `[IPAI]` 주석은 "무엇을 하는지"가 아니라 **"왜 존재하는지"**를 설명한다.

### 원칙

- `MUST` 모든 주석은 **한국어**로 작성한다.
- `MUST` 비전공자가 읽었을 때 한 문장으로 이해 가능한 수준으로 쓴다.
- `SHOULD` 전문 용어는 괄호 안에 쉬운 말로 부연한다.
  - 예: `JWT (로그인 상태를 증명하는 암호화된 열쇠)`
  - 예: `debounce (연속 입력 중 마지막 것만 처리해서 API 과호출 방지)`
- `MUST` 규칙 태그 `[PROD]`, `[ENG]` 등을 사용해 결정의 근거를 추적 가능하게 한다.

### 주석 위치별 규칙

| 위치 | 등급 | 규칙 |
|------|------|------|
| 파일 최상단 | `MUST` | 이 파일의 역할 + 속한 스킬/레이어 한 줄 |
| 함수/메서드 | `MUST` | 입력 → 출력 + 왜 이 함수가 필요한지 |
| 조건문 (`if`) | `SHOULD` | 왜 이 조건을 체크하는지 |
| 상수/설정값 | `MUST` | 왜 이 값인지, 어디서 왔는지 |
| 복잡한 로직 | `MUST` | 3줄 이상의 계산·변환은 반드시 |
| 임시 코드 | `MUST` | `// TODO: [담당자] 이유` 형식 |
| 규칙 위반 예외 | `MUST` | `// [태그] 예외 이유` 형식 |

### 좋은 주석 vs 나쁜 주석

```typescript
// ❌ 코드를 그대로 번역한 주석 (없어도 되는 주석)
const MAX_DAILY_REQUESTS = 50; // 50으로 설정

// ✅ 왜 50인지 설명하는 주석 + 관련 시스템 연결
// [COST] 하루 50회 초과 시 API 예산을 넘어섬 — token-watchdog과 임계치 동기화 필요
const MAX_DAILY_REQUESTS = 50;

// ✅ JSDoc 스타일 — 함수가 공개 API라면 항상 작성
/**
 * 사용자의 일일 API 요청 한도를 초과했는지 확인한다.
 * 초과 시 Token Watchdog에 알림을 보내고 요청을 차단한다.
 * [COST][SEC]
 *
 * @param userId - 확인할 사용자 ID
 * @returns 한도 초과 여부 (true: 차단, false: 허용)
 */
async function checkDailyLimit(userId: string): Promise<boolean> { ... }
```

### 주석 금지 사항 `[MUST]`

- 코드를 그대로 한국어로 번역한 주석
- `// 함수 시작`, `// 변수 선언` 같은 당연한 설명
- 영어 주석
- 작성자 이름·날짜 (git blame으로 확인 가능)
- 주석 처리된 죽은 코드 — 필요 없으면 완전히 삭제한다

## 8. 에러 처리 표준

> `[ENG]` 에러는 숨기지 않는다. 발생 위치에서 명확히 잡고, 사용자에게는 이해할 수 있는 메시지를 보여준다.

```typescript
// ❌ 에러를 삼켜버리는 패턴 — 문제가 생겨도 아무도 모름
try {
  await processPayment();
} catch (e) {
  console.log(e); // 로그만 남기고 끝
}

// ✅ [ENG] 올바른 에러 처리
try {
  await processPayment();
} catch (error) {
  // [COST] Token Watchdog에 에러 자동 보고 — n8n 연동으로 인사이트 리포트에 집계됨
  await reportError({ error, caller: 'processPayment', userId });

  if (error instanceof NetworkError) {
    // 네트워크 문제는 재시도 가능하다고 알려줌
    showToast('연결이 불안정합니다. 잠시 후 다시 시도해주세요.', 'warning');
  } else {
    // 예상 못한 에러는 구체적인 내용 없이 안내만 (내부 정보 노출 방지)
    showToast('처리 중 문제가 발생했습니다. 고객센터에 문의해주세요.', 'error');
  }

  throw error; // 상위 레이어도 인지할 수 있게 다시 던짐
}
```

### 에러 코드 네이밍 `[ENG][MUST]`

```typescript
// 형식: {도메인}_{행위}_{결과} — 모두 대문자, 언더스코어 구분
'AUTH_TOKEN_EXPIRED'       // 인증 토큰이 만료됨
'PAYMENT_AMOUNT_INVALID'   // 결제 금액이 유효하지 않음
'USER_PERMISSION_DENIED'   // 사용자 권한 없음
'API_RATE_LIMIT_EXCEEDED'  // API 호출 횟수 초과
```

## 9. 환경·보안 규칙

> `[SEC][MUST]` 보안 사고는 되돌리기 어렵다. 아래 규칙은 예외가 없다.

### 9-1. 환경변수 보호

```typescript
// [SEC][MUST] .env 파일은 사령관의 명시적 승인 없이 절대 덮어쓰지 않는다
// .env.example 에만 키 이름을 남기고, 실제 값은 절대 커밋하지 않는다

// ❌ 금지: 코드에 비밀 키 하드코딩
const apiKey = 'sk-ant-api03-...';

// ✅ [SEC] 환경변수로만 접근
const apiKey = process.env.ANTHROPIC_API_KEY;
if (!apiKey) throw new Error('ANTHROPIC_API_KEY 환경변수가 설정되지 않음');
```

### 9-2. Mock 데이터 환경 분리

```typescript
// [SEC][MUST] Mock 데이터는 테스트 환경에서만 사용한다
// DEV·PROD 환경에서 mock을 쓰면 실제 문제를 숨기고 보안 구멍이 생김

// ❌ 금지: DEV 환경에서 mock 유저로 인증 우회
if (process.env.NODE_ENV === 'development') {
  return mockUser; // 실제 인증 로직을 건너뜀 — 절대 금지
}

// ✅ [SEC] DEV에서도 실제 인증 흐름을 그대로 사용
// 테스트에서만 jest.mock() 또는 msw로 처리
```

### 9-3. 입력값 검증 `[SEC][MUST]`

## 10. Git · PR 규칙

> `[IPAI]` 커밋 메시지와 PR은 미래의 나와 팀원에게 보내는 편지다.

### 10-1. 커밋 메시지 `[MUST]`

무엇을 왜 했는지 한국어로 짧게 쓴다.

```
예시:
결제 금액 서버 검증 추가
토큰 만료 리다이렉트 버그 수정
PaymentButton 로직 Hook으로 분리
보안 스킬 추가
```

**커밋 원자성 `[MUST]`**
- 하나의 커밋 = 하나의 이유. 두 가지 변경이 있으면 두 번 커밋한다.
- `wip`, `임시`, `수정중`, `테스트` 같은 커밋은 main에 절대 올리지 않는다.
- 커밋 전 `git diff --staged`로 의도치 않은 파일이 포함되지 않았는지 확인한다.

### 10-2. 브랜치 전략 `[MUST]`

```
브랜치 네이밍:
feature/<기능명>   — 새 기능 개발  (예: feature/payment-webhook)
fix/<버그명>       — 버그 수정     (예: fix/auth-redirect-loop)
hotfix/<이슈명>    — 긴급 수정     (예: hotfix/api-key-expired)
refactor/<대상>    — 리팩토링      (예: refactor/payment-hook)
docs/<문서명>      — 문서 작업     (예: docs/api-spec-update)
```

- `main` 브랜치에 직접 push 금지. `[MUST]`
- 이미 공유된 브랜치에 force push 금지. `[MUST]`
- 작업 완료된 브랜치는 merge 후 즉시 삭제한다. `[SHOULD]`

### 10-3. Merge 전략 `[SHOULD]`

- **feature → main:** Squash merge — 작업 브랜치의 지저분한 커밋을 하나로 정리
- **hotfix → main:** Merge commit — 긴급 수정은 히스토리에 명확하게 남긴다
- rebase는 로컬에서만 사용한다. 공유 브랜치에서 rebase 금지.

### 10-4. Pre-commit Hook `[SHOULD]`

커밋 전 자동으로 아래를 실행한다. 하나라도 실패하면 커밋이 차단된다.

```bash
# .husky/pre-commit 예시
# [ENG][MUST] 커밋 전 자동 품질 검사 — 손으로 ICHECK 돌리는 것보다 신뢰도 높음
npm run lint        # ESLint 통과
npm run type-check  # TypeScript 에러 0개
```

### 10-5. PR 작성 및 안티패턴 규칙 `[SHOULD]`

- PR 제목은 커밋 메시지 형식과 동일하게 작성한다.
- 본문에는 "지금 이 코드가 무엇을 하는지"를 설명한다. 폐기된 접근 방식은 쓰지 않는다.
- **PR 및 안내 금칙어:** `critical`, `crucial`, `comprehensive`, `robust`, `elegant`, `seamless`, `powerful`
  - 이런 단어 없이 가치를 설명할 수 없다면 설명이 부족한 것이다.
- **코드 안티패턴 경고:**
  - `// TODO` 명시 없이 "임시방편"으로 짜둔 코드 방치 금지.
  - 빈번한 오버엔지니어링(Premature Abstraction): 한 번만 쓰이는 함수를 유틸/훅으로 미리 빼내는 판단 금지.

### 10-6. 롤백 절차 `[MUST]`

프로덕션에서 문제 발생 시:

```bash
# 1단계: 즉시 이전 커밋으로 revert (히스토리 보존)
# [ENG][MUST] reset --hard 금지 — 히스토리가 사라지면 원인 추적 불가
git revert <문제-커밋-해시>
git push origin main

# 2단계: hotfix 브랜치에서 원인 수정 후 재배포
git checkout -b hotfix/<이슈명>
```

- `git reset --hard`로 공유 브랜치 히스토리를 지우는 것 **절대 금지**. `[MUST]`
- 롤백 후 반드시 원인을 파악하고 계획.md에 기록한다. `[SHOULD]`

## 12. 파일·폴더 네이밍 규칙

> `[ENG]` 파일 이름만 봐도 역할을 알 수 있도록 일관성을 유지한다.

| 종류 | 규칙 | 예시 |
|------|------|------|
| 컴포넌트 파일 | PascalCase | `PaymentButton.tsx` |
| Hook 파일 | camelCase + `use` 접두사 | `usePayment.ts` |
| 유틸 함수 파일 | camelCase | `formatCurrency.ts` |
| 타입 정의 파일 | camelCase + `.types` | `payment.types.ts` |
| 상수 파일 | camelCase + `.constants` | `payment.constants.ts` |
| 테스트 파일 | 원본 파일명 + `.test` | `PaymentButton.test.tsx` |
| 스타일 파일 | 원본 파일명 + `.module` | `PaymentButton.module.css` |
| 폴더명 | kebab-case | `user-profile/`, `payment-flow/` |

```
features/
└── payment/
    ├── api/
    │   ├── payment.api.ts          // 서버와 통신하는 함수들
    │   └── payment.api.test.ts     // API 함수 테스트
    ├── model/
    │   ├── payment.types.ts        // 타입 정의
    │   ├── payment.constants.ts    // 상수 (예: MAX_RETRY = 3)
    │   └── usePayment.ts           // 비즈니스 로직 Hook
    ├── ui/
    │   ├── PaymentButton.tsx       // UI 컴포넌트
    │   └── PaymentButton.test.tsx  // UI 테스트
    └── index.ts                    // 외부 공개 진입점 (배럴 엑스포트)
```
