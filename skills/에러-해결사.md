# 에러 해결사 스킬 (Error Solver Skill)

> **담당:** 개발부
> **Tier:** 2 (Claude Sonnet)
> **Skill ID:** `catcher`

## Purpose

에러 발생 시 대상 워크스페이스 디렉토리만 격리해서 원인을 추적하고 수정안을 제시한다.
다른 프로덕트의 코드를 읽지 않으므로 컨텍스트 오염 없이 토큰 비용을 최소화한다.
에러 메시지 → 원인 파악 → 수정 → 검증까지 한 사이클에 완료하는 것이 목표.

## Trigger

- `@error solve {workspace}` — 지정 워크스페이스에서 발생한 에러 추적 및 수정
  - 예: `@error solve products/claimit/apps/api`
  - 예: `@error solve products/haul`
  - 예: `@error solve packages/shared`

## Input

```json
{
  "workspace": "분석할 디렉토리 경로 (필수)",
  // 예: "products/claimit/apps/api", "products/haul", "packages/ui"
  // 이 경로 밖의 파일은 절대 읽지 않는다 — 격리 원칙

  "error_message": "에러 메시지 전문 (필수)",
  // 터미널·브라우저 콘솔·서버 로그에서 복붙

  "error_type": "build | runtime | type | test | network",
  // build:   컴파일·번들링 에러 (TypeScript, Next.js build 등)
  // runtime: 실행 중 발생 (서버 크래시, unhandled promise 등)
  // type:    타입 불일치 (TS2345 등)
  // test:    테스트 실패
  // network: API 호출 실패, CORS, timeout 등

  "context": "에러 발생 상황 간단 설명 (선택)",
  // 예: "결제 완료 후 리다이렉트 시 발생", "npm run build 실행하면 나옴"

  "recent_changes": "최근에 수정한 파일/기능 (선택)"
  // 예: "payment.service.ts 수정", "node_modules 재설치 후 발생"
}
```

## Output

1. **에러 원인 분석** — 에러가 발생한 파일·라인·이유 명시
2. **수정 코드** — Before / After 형태로 즉시 적용 가능한 코드 제시
3. **검증 명령어** — 수정 후 에러가 해결됐는지 확인하는 커맨드 (타입체크, 빌드, 테스트)
4. **재발 방지 메모** — 같은 에러가 다시 발생하지 않도록 주의할 점

## Process

```
1. 격리 선언
   └─ {workspace} 디렉토리 외부 파일 읽기 금지
   └─ 단, packages/shared는 타입 확인 목적으로만 허용

2. 에러 메시지 파싱
   └─ 파일 경로·라인 번호 추출
   └─ 에러 코드 분류 (TS, ESLint, NestJS, Next.js 등 프레임워크별)

3. 에러 발생 지점 추적
   └─ 해당 파일 읽기
   └─ 에러 발생 라인 전후 20줄 분석
   └─ 의존성 파일(import 경로)까지 순차적으로 추적 (최대 3 depth)

4. 최근 변경 이력 확인
   └─ git diff HEAD (변경된 파일 목록)
   └─ 에러와 관련된 최근 커밋 내용 확인

5. 근본 원인 판별
   └─ 타입 불일치 / null 참조 / 환경변수 누락 / 의존성 버전 충돌 등 분류

6. 수정안 제시 + 검증 명령어 제공

7. ICHECK 대상 파일 지정 (수정 후 자동으로 품질 점검 범위 제한)
```

## 격리 원칙 `[ENG][MUST]`

```
✅ 읽어도 되는 것:
  - {workspace}/** 내 모든 파일
  - packages/shared/** (공유 타입 확인 목적)
  - 루트 package.json, tsconfig.base.json (버전·설정 확인)
  - git log / git diff (변경 이력 확인)

❌ 읽으면 안 되는 것:
  - 다른 products/** 코드
  - node_modules/** (버전만 package.json에서 확인)
  - .env 파일 (값이 아닌 키 이름만 확인)
```

## 자주 발생하는 에러 패턴

```
[TypeScript]
- TS2345: 타입 불일치 → 공유 타입 import 경로 확인, @claimit/shared vs 직접 정의
- TS2307: 모듈 찾을 수 없음 → tsconfig paths, barrel export 확인
- TS2339: 프로퍼티 존재하지 않음 → 타입 정의 vs 실제 DB 스키마 불일치

[NestJS]
- Nest can't resolve dependencies → 모듈에 provider 등록 누락
- Cannot read properties of undefined (reading 'get') → ConfigService inject 전 생성자 접근
- No metadata found → @Injectable(), @Controller() 데코레이터 누락

[Next.js]
- hydration mismatch → 서버/클라이언트 렌더 결과 불일치
- NEXT_REDIRECT → try-catch 안에서 redirect() 호출 (catch 밖으로 이동)
- Dynamic server usage → force-dynamic 미설정 또는 cookies() 정적 경로 사용

[환경변수]
- undefined 반환 → .env 파일 위치, NEXT_PUBLIC_ 접두사, ConfigService getOrThrow 확인
```

## Constraints

- 격리 원칙 위반 금지 — 다른 workspace 파일 읽지 않음 `[ENG][MUST]`
- 수정은 에러와 직접 관련된 파일만 — 작동하는 코드 건드리지 않음 `[IPAI]`
- 수정 후 반드시 타입체크 커맨드 제공 `[ENG]`
- 환경변수 값 직접 출력 금지 — 키 이름만 언급 `[SEC]`

## Skill Chaining

- **downstream:**
  - `ICHECK` — 수정 완료 후 품질 검토
  - `@sec check {경로}` — 에러 원인이 보안 취약점인 경우

## Hard Gates

| ID | 조건 | FAIL 기준 |
|----|------|-----------|
| G1 | 원인 특정 | 로그 재현 조건과 근본 원인이 분리되지 않음 |
| G2 | 수정 범위 | 에러와 무관한 파일 변경을 제안함 |
| G3 | 검증 명령 | 수정 후 실행할 타입체크/테스트 명령이 없음 |

## Soft Findings

| ID | 조건 | 조치 |
|----|------|------|
| S1 | 재발 방지 | 같은 에러를 막는 테스트/가드 제안 |
| S2 | 문서화 | 운영자가 알아야 할 원인을 짧게 기록 |
