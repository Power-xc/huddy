# 엣지케이스 헌터 스킬 (Edge-case Hunter Skill)

> **담당:** 테스트부
> **Tier:** 1 (Gemini Flash) → 복합 시나리오 시 Tier 2
> **Skill ID:** `edge`

## Purpose

기능 명세서 또는 코드를 입력받아 발생 가능한 예외 상황(edge case)에 대한 테스트 시나리오를 자동 생성한다.
TestSprite(외부 QA 자동화 도구)와 연동하여 24시간 자율 QA를 지원한다.

## Trigger

- 새로운 피처의 SPEC.md 작성 완료 시
- PR 생성 시 자동 실행
- `@edge hunt {feature_name}` 명령 호출 시
- Code Architect Skill의 generate 모드 완료 후 자동 체이닝

## Input

```json
{
  "mode": "spec_based | code_based | api_based",
  // spec_based: 기획서를 읽고 예외 상황 예측
  // code_based: 실제 코드를 분석해서 취약한 분기점 탐색
  // api_based: API 엔드포인트 기준으로 잘못된 입력값 케이스 탐색

  "source": "SPEC.md 경로 또는 분석할 코드 파일 경로",
  "feature_name": "테스트할 기능 이름",

  "focus": ["auth", "validation", "concurrency", "network", "boundary", "permission"],
  // 집중적으로 테스트할 카테고리 선택
  // auth: 로그인/인증 관련  |  validation: 입력값 검증
  // concurrency: 동시에 여러 요청이 들어올 때  |  network: 네트워크 오류 상황
  // boundary: 최대/최소값 경계  |  permission: 권한 없는 접근 시도

  "output_format": "jest | vitest | playwright | cypress",
  // 테스트 코드를 어떤 형식으로 출력할지 (프로젝트에서 쓰는 테스트 도구에 맞춤)

  "severity_threshold": "critical | high | medium | all"
  // 어느 수준 이상의 위험도만 뽑을지
  // critical: 서비스 중단급  |  high: 데이터 손상급  |  all: 모두 포함
}
```

## Process

```
1. 소스 파싱
   ├─ spec_based: SPEC.md에서 기능 요구사항 추출
   │   // "이런 기능이 있다면, 이런 예외가 생길 수 있다"는 방식으로 접근
   ├─ code_based: AST 분석으로 분기/조건문/에러 핸들링 추출
   │   // AST: 코드를 나무 구조로 분해해서 분석하는 방식
   │   // if/else, try/catch가 있는 곳 = 예외가 생길 수 있는 곳
   └─ api_based: API 엔드포인트별 입력 파라미터 추출
       // "이 API에 이상한 값을 넣으면 어떻게 되는가?" 체크

2. Edge Case 카테고리별 패턴 매칭
   ├─ auth: 토큰 만료, 권한 에스컬레이션, 동시 세션
   │   // 토큰(로그인 증명서)이 만료됐을 때 / 일반 유저가 관리자 권한 시도
   ├─ validation: 경계값, null/undefined, 특수문자, SQL injection
   │   // null/undefined: 값이 없을 때  |  SQL injection: DB를 해킹하는 특수 문자열
   ├─ concurrency: 레이스 컨디션, 데드락, 중복 요청
   │   // 레이스 컨디션: 두 작업이 동시에 같은 데이터를 수정할 때 충돌
   │   // 데드락: 서로가 서로를 기다리며 멈춰버리는 상황
   ├─ network: 타임아웃, 재시도, 오프라인 상태
   ├─ boundary: 최대/최소값, 빈 배열, 페이지네이션 끝
   │   // 페이지네이션 끝: 마지막 페이지 다음 페이지를 요청하면?
   └─ permission: RLS 우회 시도, 타 사용자 데이터 접근
       // RLS: Row Level Security — DB에서 내 데이터만 볼 수 있게 막는 보안 규칙

3. severity 등급 자동 분류
4. 테스트 코드 생성 (output_format 기준)
5. coverage_map 계산 및 미커버 영역 경고
   // "이 부분은 아직 테스트가 없습니다" 알림
6. TestSprite(외부 QA 자동화 도구) 연동 시 자동 실행 큐 등록
   // 외부 QA 자동화 도구가 테스트를 실행해서 결과를 알려줌
```

## Example

**Input:**
```json
{
  "mode": "spec_based",
  "source": "./docs/SPEC_auth.md",
  "feature_name": "login",
  "focus": ["auth", "validation"],
  "output_format": "vitest",         // 이 프로젝트의 테스트 도구
  "severity_threshold": "high"       // high 이상만 추출 (critical + high)
}
```

**Output (test_code 발췌):**
```typescript
// 로그인 기능의 예외 상황 테스트 모음
describe('Login Edge Cases', () => {

  // EDGE-001: 이메일이 너무 길 때 서버에 요청조차 안 가야 함
  // (255자 초과 이메일은 DB에 저장 자체가 불가능하므로 미리 막아야 함)
  it('EDGE-001: should reject email exceeding 255 characters', async () => {
    const longEmail = 'a'.repeat(256) + '@test.com'; // 256자짜리 이메일 생성
    const result = await login({ email: longEmail, password: 'valid123!' });

    expect(result.error).toBe('EMAIL_TOO_LONG');  // 이 에러 코드를 반환해야 함
    expect(result.statusCode).toBe(400);           // 400 = 잘못된 요청
  });

  // EDGE-002: 로그인 유지 토큰이 만료됐을 때 로그인 화면으로 보내야 함
  // (토큰 만료를 무시하면 보안 구멍이 생김)
  it('EDGE-002: should handle expired refresh token gracefully', async () => {
    mockTokenExpired(); // 토큰이 만료된 상황을 가짜로 만들어서 테스트
    const result = await refreshSession();

    expect(result.redirectTo).toBe('/login');    // 로그인 페이지로 이동해야 함
    expect(result.clearStorage).toBe(true);      // 만료된 토큰을 삭제해야 함
  });
});
```

## Constraints

- critical 시나리오는 반드시 테스트 코드 포함 `[ENG]`
- SQL injection / XSS 패턴은 항상 자동 포함 (focus 무관) `[SEC]`
  // SQL injection: DB를 공격하는 코드  |  XSS: 남의 브라우저에서 스크립트 실행
  // 보안 공격은 focus 설정과 관계없이 무조건 체크
- 생성된 테스트는 독립 실행 가능해야 함 (외부 의존성 mock 처리) `[ENG]`
  // mock: 실제 DB나 API 대신 가짜를 쓰는 것 — 빠르고 안정적으로 테스트 가능
- 시나리오 ID는 글로벌 유니크 (`EDGE-{sequential}`) `[SPEC]`

## Skill Chaining

- **upstream:** Code Architect Skill (generate 완료 후 자동 체이닝)
- **downstream:** TestSprite(외부 QA 자동화 도구) 자율 QA

## Hard Gates

| ID | 조건 | FAIL 기준 |
|----|------|-----------|
| G1 | 치명 시나리오 | 인증, 결제, 권한, 데이터 손상 중 관련 케이스가 누락됨 |
| G2 | 재현 가능성 | 시나리오에 precondition, steps, expected outcome이 없음 |
| G3 | 우선순위 | critical/high 구분 없이 나열만 함 |

## Soft Findings

| ID | 조건 | 조치 |
|----|------|------|
| S1 | 자동화 후보 | 반복 가능한 시나리오는 테스트 코드 후보로 표시 |
| S2 | 관찰 지표 | 장애 감지를 위한 로그/메트릭 제안 |
